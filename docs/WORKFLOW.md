# WORKFLOW.md — カード追加の運用手順（Cowork/Claude向け）

このドキュメントは、Scholar Speakに新しい単語カードを追加するときの手順書です。
**将来のClaudeセッション（Cowork）が、このドキュメントだけを読んでそのまま作業できること**を目的にしているため、抽出方針・スキーマ・品質基準を省略せず全文で記載しています。

対象読者: この運用を実行するClaude（今後のセッション）、および運用フローを確認したい学習者本人。

---

## 0. 前提: 誰のためのアプリか

- ユーザーは、大学院レベルの英語が必要な学習者。学際的な学び（既存の学問分野の枠組みにとらわれず、複数分野を横断する研究）に取り組んでいることを想定している。
- 大学院レベルの英単語・学術表現に苦戦しており、英語でのディスカッションとプレゼンテーションに参加する必要がある。
- 単語を単体で覚えるのではなく、**授業やゼミで実際に使われた文脈**とセットで、かつ**プレゼン・議論でそのまま暗唱できるセンテンス**として覚えたい、というのがこのアプリの核心的な思想。

したがって、カードを追加するときは常に「この単語・表現を学習者が実際に自分の研究発表や議論で使うとしたら、どんな一文になるか」を意識すること。

---

## 1. 全体フロー

1. 学習者が、大学院の授業のスライド・ゼミでの会話記録・論文の抜粋・教員のフィードバックメモなどのテキストをCowork（Claude）に渡す。
2. Claudeがそのテキストから、TOEFL〜大学院レベルの単語・学術的言い回し（academic phrase）を抽出する。
3. Claudeが抽出した語・表現ごとに、後述のスキーマに厳密準拠したエントリ（entry）を生成する。
   - このとき、**`context_quote`には授業・ゼミでの実際の使用文を必ず入れる**（一字一句そのままか、要約せずに使われた文をできる限り正確に再現する）。
   - **新しい授業・ゼミ単位で新しいdeckを作成する**（`meta.decks`に追加し、各entryの`deck`にそのIDを指定する）。deck idは「授業名+日付」を推奨（例: `design-pedagogy-2026-07-06`）。
4. 生成したentriesを既存の `data/words.json` にマージする（`meta.entries`配列に追記。既存のentryとidが重複しないよう注意）。
5. `meta.updated` を実行日の日付（`YYYY-MM-DD`）に更新する。
6. `python3 -m json.tool data/words.json > /dev/null` などでJSONとして妥当か検証する。
7. `git add`, `git commit`, `git push` でリポジトリに反映する。
   ```bash
   git add data/words.json
   git commit -m "Add deck: <授業名> (<日付>) - N words"
   git push
   ```
8. 学習者がアプリを開き、設定画面の「データ再読込」をタップして最新の`words.json`を読み込む（GitHub Pagesの反映には数分かかることがある）。

## 2. データスキーマ全文（`data/words.json`）

このファイル全体は次の構造を持つ、単一のJSONオブジェクトである。

```json
{
  "meta": {
    "version": 1,
    "updated": "2026-07-06",
    "decks": [
      { "id": "seed", "name": "スターターデッキ", "source": "サンプル", "date": "2026-07-06" },
      { "id": "design-pedagogy-2026-07-06", "name": "デザイン教育論 ゼミ (7/6)", "source": "デザイン教育論 第10回スライド", "date": "2026-07-06" }
    ]
  },
  "entries": [
    {
      "id": "paradigm",
      "type": "word",
      "deck": "seed",
      "term": "paradigm",
      "ipa": "/ˈpærədaɪm/",
      "pos": "noun",
      "level": "TOEFL/C1",
      "meaning_ja": "パラダイム、（思考の）枠組み",
      "gloss_en": "a typical model or framework of ideas",
      "etymology": "ギリシャ語 para-(そばに) + deigma(示すもの)。日本語で語源を丁寧に。",
      "mnemonic": "日本語での覚え方・既知知識との接続。苦戦する人向けに具体的に。",
      "context_quote": { "en": "授業で実際に使われた一文(あれば)", "ja": "その和訳", "source": "授業名/スライド 2026-07-06" },
      "examples": [
        { "en": "理解用例文", "ja": "和訳", "scene": "discussion" }
      ],
      "sentences": [
        { "en": "プレゼン/議論でそのまま使える暗唱用センテンス", "ja": "和訳", "scene": "presentation" }
      ],
      "synonyms": ["framework", "model"],
      "paraphrases": ["a shared way of thinking about ..."],
      "antonyms": ["anomaly"],
      "collocations": ["paradigm shift", "dominant paradigm"],
      "register": "academic",
      "tags": ["academic", "education"],
      "source": "seed"
    }
  ]
}
```

### フィールド仕様

**`meta`**

| フィールド | 型 | 説明 |
|---|---|---|
| `version` | number | スキーマバージョン。基本的に`1`のまま変更しない。 |
| `updated` | string (`YYYY-MM-DD`) | ファイル最終更新日。マージ作業のたびに実行日に更新する。 |
| `decks` | array | デッキ定義のリスト。 |

**`meta.decks[]`（デッキ定義）**

| フィールド | 型 | 説明 |
|---|---|---|
| `id` | string | デッキの一意なID。半角英数とハイフンのみ推奨。**新しい授業回・ゼミごとに新規作成**（例: `design-pedagogy-2026-07-06`、`systems-seminar-2026-07-13`）。 |
| `name` | string | 日本語の表示名（例: 「デザイン教育論 ゼミ (7/6)」）。 |
| `source` | string | 出典（授業名・資料名など、人間が読んで分かる形）。 |
| `date` | string (`YYYY-MM-DD`) | その授業・ゼミが行われた日付。 |

**`entries[]`（各カード）**

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| `id` | string | 必須 | 一意なID。半角英数とハイフンのみ。既存IDと重複させない（同じ単語が別deckで再出現する場合は `word-deckid` のようにサフィックスを付けて区別する）。 |
| `type` | `"word"` \| `"phrase"` | 必須 | `word`=単語、`phrase`=学術的言い回し（複数語からなる定型表現）。 |
| `deck` | string | 必須 | 所属する`meta.decks[].id`。 |
| `term` | string | 必須 | 見出し語・表現そのもの。`phrase`は文の形（例: `"What I find compelling is ..."`）で構わない。 |
| `ipa` | string \| null | 任意 | IPA発音記号。`phrase`型では省略・`null`可。 |
| `pos` | string | 必須 | 品詞（`noun`, `verb`, `adjective`, `adverb`など）。`phrase`型は`"phrase"`とする。 |
| `level` | string | 必須 | 難易度レベル。`"TOEFL/C1"` または `"Graduate"` を基本とする（TOEFL相当〜大学院レベルの範囲であることを示す）。 |
| `meaning_ja` | string | 必須 | 日本語の意味。丁寧で分かりやすい説明にする（後述「品質基準」参照）。 |
| `gloss_en` | string | 必須 | 平易な英語での定義（英英辞典の初級者向け定義くらいの語彙レベル）。 |
| `etymology` | string \| null | 任意 | 語源の日本語での分解解説。`phrase`型は`null`可。 |
| `mnemonic` | string | 必須 | 日本語での具体的な記憶フック（後述「品質基準」参照）。 |
| `context_quote` | object \| 省略可 | **原則必須（授業由来の場合）** | `{ "en": "...", "ja": "...", "source": "..." }`。**その単語・表現が実際に使われた授業・会話・資料中の一文**。シードデータなど元テキストが存在しない場合はフィールド自体を省略してよい。 |
| `examples` | array（2件以上推奨） | 必須 | 理解用の例文。各要素は `{ "en", "ja", "scene" }`。 |
| `sentences` | array（**2件以上**） | 必須 | プレゼン・議論でそのまま暗唱して使えるセンテンス。各要素は `{ "en", "ja", "scene" }`。 |
| `synonyms` | string[] | 必須（無ければ`[]`） | 類義語・類義表現。 |
| `paraphrases` | string[] | 必須（無ければ`[]`） | 難しい語→平易な言い換え表現。 |
| `antonyms` | string[] | 必須（無ければ`[]`） | 反意語。無い場合は空配列。 |
| `collocations` | string[] | 必須（無ければ`[]`） | よく使われる連語。 |
| `register` | string | 必須 | 通常`"academic"`。 |
| `tags` | string[] | 必須 | 分類タグ（例: `["academic", "education"]`, `["design", "research"]`）。 |
| `source` | string | 必須 | データの由来。シードデータは`"seed"`、授業由来は`deck`のidと同じ値、または資料名。 |

`examples`/`sentences`内の`scene`は次のいずれか: `"presentation"` | `"discussion"` | `"qa"` | `"reading"`。

## 3. 抽出方針（どの語をカード化するか）

渡されたテキスト（スライド・会話記録など）から、次の基準で語・表現を選ぶ。

1. **難易度**: TOEFL上級〜大学院レベル（C1〜C2相当）。中学・高校レベルの基礎語彙は対象外。
2. **頻度と重要度**: そのテキストの中で意味的に重要な語（キーコンセプト、議論のキーワード）を優先する。単なる固有名詞や、明らかに一度きりの専門用語（他分野の非常に狭い専門語）は無理に含めない。
3. **再利用可能性**: 今後も学習者が別の授業・議論で使い回せそうな語・表現を優先する（例: `epistemology`, `scaffold`, `articulate` のような、分野を超えて汎用性の高い学術語）。
4. **学術的言い回し（`phrase`型）**: 単語だけでなく、ディスカッション・プレゼンで使える定型表現（例: `"What I find compelling is ..."`, `"To build on that point, ..."`, `"This raises the question of whether ..."`, `"I'd like to push back on that a little."` など）も積極的に抽出する。1回の抽出で少なくとも1〜2個は含めることを推奨。
5. 1回のマージ作業で追加するカード数の目安: **10〜20件程度**（多すぎると復習負荷が過剰になるため、テキストが長い場合は特に重要な語に絞る）。

## 4. パーソナライズ方針

- **`meaning_ja`**: 丁寧語で、専門用語を使う場合は補足を添える。大学院生が読んでも「易しい日本語」で理解できるレベルにする。
- **`gloss_en`**: 中学〜高校レベルの平易な英単語だけで書く英英定義。難しい語で難しい語を説明しない。
- **`etymology`**: 日本語で、語根（ラテン語・ギリシャ語など）を分解して説明する。分解できない語やphrase型はフィールド省略または`null`。
- **`mnemonic`**: **日本語での具体的な記憶フック**。以下のいずれかのパターンを使うと効果的:
  - 音の響きが似ている日本語・カタカナ語と結びつける。
  - 語源の分解を使ったイメージ化（例: 「con(共に)+note(印)」）。
  - 学習者自身の専攻・研究テーマ（学際的な学び、デザイン、教育など、プロファイルがあればそれ）に関連する具体的なシーンをイメージさせる。
  - 単なる「〜という意味です」の言い換えではなく、覚えるための「フック（引っかかり）」を必ず作ること。
- **`examples`/`sentences`**: できる限り以下の文脈に寄せる。
  - 大学院の授業、ゼミでのディスカッション
  - 学習者の専攻に関するプロジェクト・ワークショップ
  - 分野を横断する研究プロジェクト
  - 研究発表・プレゼンテーションの場面
  - 一人称（"I", "we", "my thesis", "our project"など）を使い、学習者自身が話しているかのような一文にする。
- **`context_quote`**: 授業由来のカードを作る場合、**必ず元テキストにある実際の文（またはそれに最も近い形）を入れる**。要約や言い換えをせず、実際に使われたニュアンスを保持すること。出典（`source`）には授業名と日付を明記する。
- **`paraphrases`**: 「難しい語 → 平易な言い換え」の方向で書く。学習者が本番でその単語をド忘れしても、平易な言葉で同じ意味を伝えられるようにするための代替表現。

## 5. 品質基準（チェックリスト）

新しいentryをマージする前に、以下を確認すること。

- [ ] `id`が既存entryと重複していない。
- [ ] `type`が`"word"`か`"phrase"`のいずれかで、実態と一致している。
- [ ] `deck`が`meta.decks[]`に実在するIDである（新規deckなら`meta.decks`に追加済み）。
- [ ] `level`が`"TOEFL/C1"`または`"Graduate"`（同等の表記）になっている。
- [ ] `meaning_ja`が丁寧で分かりやすい日本語になっている。
- [ ] `gloss_en`が平易な英語（難語を難語で説明していない）になっている。
- [ ] `mnemonic`が具体的で、学習者の文脈や語源に基づいたフックになっている（一般論で終わっていない）。
- [ ] `sentences`が**2件以上**あり、いずれもプレゼン・議論でそのまま暗唱して使える完結した一文になっている。
- [ ] `examples`が2件以上ある。
- [ ] 授業由来のentryには`context_quote`が入っており、`source`に授業名と日付が明記されている（シードデータなど元テキストがない場合は省略可）。
- [ ] `synonyms` / `paraphrases` / `antonyms` / `collocations`が空配列であっても必ずキー自体は存在する。
- [ ] JSON全体が有効なJSONとして解析できる（下記コマンドで検証）。
- [ ] `meta.updated`を作業日の日付に更新した。

### JSON検証コマンド

マージ後、必ず以下を実行して構文エラーがないことを確認する。

```bash
python3 -m json.tool data/words.json > /dev/null && echo "OK: valid JSON"
```

エラーが出た場合は、カンマの過不足や引用符の閉じ忘れがないか確認し、修正後に再検証する。

## 6. Git操作とアプリへの反映

```bash
cd lexilab
git add data/words.json
git commit -m "Add deck: <deck.name> (<deck.date>) - <N> entries"
git push
```

pushが完了したら、GitHub Pagesへの反映を待ってから（数分程度）、学習者にアプリの設定画面で「データ再読込」を実行してもらう。反映が遅い場合はブラウザのキャッシュクリアやページの再読み込みを案内する。

## 7. 注意事項

- **index.html / css / js には触れないこと。** これらはアプリ実装側（別のClaudeセッション/エージェント）が管理する。データ担当が触れてよいのは `data/words.json`、`README.md`、`docs/`配下のみ。
- 個人が特定される可能性のある発言（教員名・他の学生の実名など）は`context_quote`にそのまま含めず、必要に応じて匿名化するか、`source`に留める（授業名・回のみ）。
- 1回のセッションで抽出する語数が多すぎる場合は、複数回の`deck`に分けるのではなく、重要度の高いものから絞り込むことを優先する（復習の認知負荷を考慮）。

## 学習者プロファイル(任意)

このアプリはパーソナライズを前提に設計されている。カード生成時に、学習者の専攻・研究テーマ・受講中の授業といった文脈を（任意で）与えると、`mnemonic`・`examples`・`sentences`をその文脈に合わせて調整できる。

- プロファイルは任意。与えられていない場合は、大学院レベルの学際的な学びを一般的な想定として例文を作る。
- プロファイルが与えられた場合は、その専攻・研究テーマ・主要教材・具体的な学習シーン（実験、研究発表、ワークショップなど）に例文を寄せる。
- いずれの場合も、一人称（"I", "we", "my thesis", "our project"など）で、学習者が実際のディスカッションやプレゼンでそのまま暗唱して使える一文にすること。
